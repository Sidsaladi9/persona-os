# sql-queries — worked example (Cadence)

> Generated by the Product Manager OS `sql-queries` skill. Demo company: Cadence.

---

**TL;DR**
- We want week-2 retention (active in days 7–14) for teams that started check-ins, bucketed by weekly signup cohort, excluding cohorts too recent to have completed the window. 🟡
- The query is a retention/cohort shape on a half-open window `[+7d, +14d)`, with a `WHERE` clause that drops any cohort whose 14-day window hasn't fully elapsed as of `now()`.
- Headline read on real numbers: week-2 retention sits around 41–46% and is trending down across recent cohorts — consistent with the week-2 drop-off we're chasing this sprint.

## Question

What share of teams that posted their first check-in in a given signup week are still active (posted at least one check-in) anywhere in the day-7-to-day-14 window after their first check-in? One row per signup week (Monday-anchored), bucketed in account-local time (US Eastern), excluding cohorts whose 14-day window has not fully closed.

## Shape

Retention/Cohort — we anchor each team on its first check-in, then measure return activity in a fixed forward window. This is the canonical "week-2 retention" N-day-window pattern (active *any* time in `[+7d, +14d)`), not day-7-exact.

## Assumptions

- Dialect: Postgres (Cadence analytics replica).
- Events table: `check_in_events`, team id: `team_id`, timestamp: `created_at` (timestamptz, stored UTC), event type: implicit — every row in this table is one posted check-in, so no `event_type` filter is needed.
- "Active in week 2" means: the team posted ≥1 check-in on any day in `[first_check_in_day + 7d, first_check_in_day + 14d)`.
- "Cohort" means: teams grouped by the Monday-anchored week of their *first ever* check-in.
- Timezone: account-local `America/New_York` (Cadence buckets days by ET so a late-evening US check-in counts on the right calendar day); grain: day for the window math, week for the cohort.
- One team can have many check-ins per day; we de-dupe to one row per team per active day before joining.

## SQL

```sql
-- PATTERN: RETENTION — N-day WINDOW [+7d, +14d) ("week-2 retention").
-- A team is "retained in week 2" if it is active ANY day in the window, NOT
-- exactly on day 7. Half-open interval [start, end) avoids double-counting the
-- boundary day. Both sides of the comparison are day-truncated in the SAME
-- timezone (ET), so a raw-timestamp anchor can never silently miss matches.
WITH first_seen AS (                         -- anchor: each team's first check-in day (ET)
  SELECT
    team_id,
    DATE_TRUNC('day', MIN(created_at) AT TIME ZONE 'America/New_York') AS first_day
  FROM check_in_events
  GROUP BY team_id
),
cohort AS (                                  -- denominator: bucket each team into a signup week
  SELECT
    team_id,
    first_day,
    DATE_TRUNC('week', first_day) AS cohort_week   -- Postgres week = ISO, Monday-anchored
  FROM first_seen
),
activity AS (                                -- one row per team per active day (ET), de-duped
  SELECT DISTINCT
    team_id,
    DATE_TRUNC('day', created_at AT TIME ZONE 'America/New_York') AS active_day
  FROM check_in_events
)
SELECT
  c.cohort_week,
  COUNT(DISTINCT c.team_id)                                   AS cohort_size,
  COUNT(DISTINCT CASE
    WHEN a.active_day >= c.first_day + INTERVAL '7 day'
     AND a.active_day <  c.first_day + INTERVAL '14 day'
    THEN c.team_id END)                                       AS retained_w2,
  ROUND(100.0 * COUNT(DISTINCT CASE
    WHEN a.active_day >= c.first_day + INTERVAL '7 day'
     AND a.active_day <  c.first_day + INTERVAL '14 day'
    THEN c.team_id END) / NULLIF(COUNT(DISTINCT c.team_id), 0), 1) AS retention_w2_pct
FROM cohort c
LEFT JOIN activity a ON a.team_id = c.team_id   -- LEFT JOIN keeps non-returning teams in the denominator
-- Exclude immature cohorts: the FULL 14-day window must have elapsed. Use the
-- window END (+14d), NOT +7d — otherwise late-window returns are silently missing
-- and the newest cohorts read artificially low. Anchor the cutoff on each team's
-- first_day-derived week; a cohort_week that started < 14 days ago can't be mature.
WHERE c.cohort_week + INTERVAL '14 day' <= DATE_TRUNC('day', now() AT TIME ZONE 'America/New_York')
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

## Result

| cohort_week | cohort_size | retained_w2 | retention_w2_pct |
|---|---|---|---|
| 2026-04-27 | 312 | 147 | 47.1 |
| 2026-05-04 | 358 | 165 | 46.1 |
| 2026-05-11 | 341 | 149 | 43.7 |
| 2026-05-18 | 327 | 138 | 42.2 |
| 2026-05-25 | 369 | 152 | 41.2 |
| 2026-06-01 | 354 | 144 | 40.7 |

> Cohorts from the weeks of 2026-06-08 onward are intentionally absent — their 14-day window has not fully closed as of 2026-06-17, so including them would understate retention.

## Sanity checks (run before trusting)

- [x] **Denominator size** — `cohort_size` per week (312–369) lines up with our known ~340 new teams/week from the signup dashboard. ✅
- [x] **People vs events** — used `COUNT(DISTINCT team_id)` everywhere; never `COUNT(*)`. We count teams, not check-ins.
- [x] **Spot-check one team** — picked `team_id = 'tm_8841'` (first check-in 2026-05-06, ET): raw `check_in_events` shows a check-in on 2026-05-14 (day 8), so it correctly lands in `retained_w2` for the 2026-05-04 cohort. ✅
- [x] **Monotonicity** — retention only moves *down* over forward windows; week-2 (~41–47%) is below our week-1 figure (~63%), as expected. A week-2 number *above* week-1 would be a bug.
- [x] **Boundaries** — half-open `[+7d, +14d)`; a day-14 check-in belongs to week 3, not week 2; timezone fixed at ET on both anchor and activity sides.
- [x] **Complete cohorts only** — `WHERE cohort_week + 14d <= now()` drops the 2026-06-08+ cohorts that haven't matured.
- [x] **Anchor alignment** — `first_day` and `active_day` are both `DATE_TRUNC('day', … AT TIME ZONE 'America/New_York')`, so the `+7d`/`+14d` comparison can't silently zero out.
- [x] **Nulls & div-by-zero** — `NULLIF(denominator, 0)` guards the percent; `LEFT JOIN` keeps the full cohort in the denominator even for teams that never returned.

## Read of the result

Each row is one signup-week cohort of teams: how many started check-ins, and how many came back at least once in days 7–14. The headline is week-2 retention drifting from 47.1% (late April) down to 40.7% (early June) — a steady erosion that matches the week-2 drop-off we're prioritizing this sprint. The assumption most likely to flip the answer is the *window definition*: if leadership means "active *exactly* on day 7" rather than "anytime in days 7–14," swap to the day-N-exact pattern and every number here drops by roughly 10–15 points.
