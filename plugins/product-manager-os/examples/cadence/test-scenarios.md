# Test Scenarios: Blocker Detector

**Spec reference:** PRD-218 "Blocker Detector v1" · Linear CAD-1042
**Author / date:** Maya Chen (PM) · 2026-06-17
**Assumptions made:**
- Detector runs on each submitted check-in (text fields: *yesterday*, *today*, *blockers*) plus the prior 5 check-ins from the same person, and writes flagged items to the manager's daily digest. No real-time UI on the check-in itself in v1.
- "At-risk work" = a model-flagged item with a confidence score 0–1; the digest shows items at score ≥ 0.60. Below 0.60 is suppressed (not stored as a surfaced flag).
- Detector reads only check-ins within the flagging team's workspace; it never reads across workspaces.
- Manager can mark a flag "Resolved" or "Not a blocker" (dismiss). Dismissals feed the model's feedback loop.
- v1 is English-only for the model; non-English text is stored verbatim and passed through but not expected to flag accurately.

**Out of scope:**
- The week-2 retention lift and the "managers act on flags" outcome metric — those are validated in analytics, not QA. Here we only assert that the `blocker_flag_surfaced`, `blocker_flag_resolved`, and `blocker_flag_dismissed` instrumentation events fire with the correct payload (`team_id`, `checkin_id`, `confidence`, `surfaced_in_digest`).
- Digest email rendering/deliverability (covered by the existing digest test suite).
- Model retraining pipeline correctness (offline; separate eval).

## 7. ML quality-bar / eval cases
**Golden set size:** N=300 labeled check-ins (200 with a real blocker, 100 clean), drawn from production check-ins across 25 teams, anonymized.
**Label source:** production-sampled, human-rated by 2 raters with a tie-breaker (inter-rater agreement reported; disagreements adjudicated by PM).
**Aggregate metric & bar:**
- Precision ≥ 0.80 and recall ≥ 0.70 over N=300 at the 0.60 threshold (precision weighted higher — a noisy digest is worse than a missed flag for week-2 trust).
- 0 surfaced flags on the 100 clean check-ins that quote sensitive content as a "blocker" incorrectly is not required, but false-positive rate ≤ 0.20.
- p95 scoring latency < 3s per check-in.
**Judging method:** automated precision/recall against human labels; latency from service metrics; safety slice human-reviewed.

> Assert the aggregate metric over the golden set, not per-row exact matches. The rows below are illustrative slices, not the pass/fail unit.

| ID | Input slice | Expected behavior | Aggregate assertion | Priority |
|---|---|---|---|---|
| ML-1 | Typical real blockers ("waiting on X", "stuck because Y", "still no answer on Z") | Flagged at ≥ 0.60 | recall ≥ 0.70 over the 200-positive slice | P0 |
| ML-2 | Clean / on-track check-ins, including ones that mention finished work or future risks that aren't blockers yet | Not flagged | precision ≥ 0.80; false-positive rate ≤ 0.20 over the 100-clean slice | P0 |
| ML-3 | Ambiguous ("might be a problem next week if the API changes") | Reasonable, consistent call — either suppressed or low-confidence; never high-confidence false alarm | hallucinated-blocker rate ≤ 0.10 on a 40-item ambiguous slice | P1 |
| ML-4 | Adversarial / injection slice (DV-5 style + venting/sarcasm not meant as blockers) | No instruction-following; sarcasm/venting not surfaced as real blockers | 0 model-driven actions from injected text; ≤ 0.15 false-positive on the sarcasm sub-slice over N=50 | P1 |
