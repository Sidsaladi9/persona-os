# user-stories — worked example (Cadence)

> Generated by the Product Manager OS `user-stories` skill. Demo company: Cadence.

---

**TL;DR**
We broke the **AI Blocker Detector** epic into 8 vertical stories sliced thin enough to fit our reduced-capacity sprint (3 engineers, one on PTO 3 days). The thinnest first slice is an eval-only detection story — we prove the model hits its precision/recall bar against a labeled set before any UI ships. Status: 🟡 at risk on timeline because the model-accuracy slice (S1) gates everything visible to teams.

---

## Epic input

> **Epic:** AI Blocker Detector
>
> Teams write async check-ins each morning ("Yesterday / Today / Blockers"). Today the "Blockers" field is free text and managers skim it manually in the digest. Many real blockers are buried in the "Today" or "Yesterday" text, never tagged, and surface too late. We want Cadence to read each check-in and flag work that looks **at risk** — stalled tasks, repeated mentions of waiting on someone, hedging language ("might slip", "still stuck on") — and surface those flags in the manager digest so leads act before the work slips a week.
>
> **Primary persona:** Engineering manager / team lead (digest reader). Secondary: the team member writing the check-in (flag accuracy + dispute).
> **Outcome:** Leads see at-risk work the day it appears, not at the weekly retro. Ties to priority (1) week-2 retention — managers who get useful signal keep their teams checking in.
> **Constraints:** Small product team (1 PM, 3 eng, 1 of whom is on PTO 3 days this sprint, 1 designer, shared data analyst). Async product — no real-time pressure, but the digest is generated once daily at the team's configured send time.

## Labeled eval set (built for this epic)

The shared data analyst (Priya) pulled and hand-labeled a golden set from anonymized historical check-ins:

| Item | Value |
|---|---|
| Dataset | `blocker-eval-v1` — 1,200 check-ins from 84 teams, opted-in to product improvement |
| Positive label | "at-risk" — confirmed blocked/slipped within 5 working days (cross-checked against later check-ins) |
| Positives | 312 (26%) |
| Negatives | 888 (74%) |
| Labelers | Priya (analyst) + 2 eng spot-check; disagreements (n=47) resolved by majority |
| Held-out test split | 240 items (20%), never seen during prompt tuning |

Targets agreed with the team: a missed real blocker (false negative) is costlier than a noisy flag a lead can dismiss, so we weight recall, but we cap noise so the digest stays trusted.

## Stories

### S1 — Detect at-risk work against the eval set (offline)

As a **product team building the Blocker Detector**,
I want the detection model to flag at-risk check-ins and hit an agreed accuracy bar against the labeled `blocker-eval-v1` set,
so that we know the signal is trustworthy before we ever put a flag in front of a lead.

**Acceptance criteria**
- Given the held-out test split of `blocker-eval-v1` (240 labeled check-ins), when the detector runs over all of them, then **recall ≥ 0.80** and **precision ≥ 0.70** on the "at-risk" label.
- Given the full eval set, when results are scored, then a confusion matrix and per-error breakdown (false positives / false negatives with the source text) are written to a run artifact the team can review.
- Given a check-in with empty or whitespace-only text, when the detector runs, then it returns "no signal" (not at-risk) and does not error.
- Given a check-in not in English (we support English-only at launch), when the detector runs, then it returns "no signal" and the item is logged as `unsupported_language`.

**Notes / edge cases**
- This is an eval-only slice: no UI, no digest change. Per the skill, a probabilistic capability proven against a fixed eval set is a legitimate first vertical slice — the verified accuracy is the value.
- AC is threshold-based, not per-instance, because detection is ~85%-correct ML behavior, not deterministic.
- If we miss the bar, the fix is prompt/model iteration on this same story — not a new story.

**Out of scope**
- Showing flags to anyone. Storing flags. Disputes. Multi-language. All deferred to later slices.

---

### S2 — Surface at-risk flags in the daily digest

As an **engineering manager reading my team's daily digest**,
I want at-risk check-ins flagged inline in the digest,
so that I see work that's slipping the day it appears instead of at the weekly retro.

**Acceptance criteria**
- Given my team has check-ins that the detector flagged at-risk, when today's digest is generated at my team's configured send time, then each flagged check-in shows a 🟡 at-risk marker with the one-line reason (e.g. "waiting on review for 3 days").
- Given a digest where no check-ins are flagged, when the digest is generated, then no Blocker Detector section appears (no empty state clutter).
- Given a check-in is flagged, when I read the digest, then I can see the team member's name, the check-in date, and the snippet of text that triggered the flag.
- Given the detector run failed for my team today, when the digest is generated, then the digest still sends with all other content and shows "Blocker Detector unavailable today" rather than blocking or omitting the digest.

**Notes / edge cases**
- Depends on S1 clearing the accuracy bar.
- Designer (Maya) owns the flag treatment in the digest; keep it skimmable — bullets, not prose.
- "at-risk" copy must use product surface language: "check-in", "digest". No internal codename in the digest itself.

**Out of scope**
- Configuring sensitivity. Disputing a flag. Anything outside the digest surface (e.g. Slack notifications).

---

### S3 — See why a check-in was flagged

As an **engineering manager**,
I want to expand a flag and see the specific phrases the detector picked up,
so that I trust the signal and know what to ask my report about.

**Acceptance criteria**
- Given a flagged check-in in the digest, when I expand it, then I see the highlighted trigger phrases (e.g. "still blocked on", "might slip to next week") within the original check-in text.
- Given a flag with multiple triggers, when I expand it, then all contributing phrases are highlighted, not just the first.
- Given the original check-in has been edited after flagging, when I expand the flag, then I see the text as it was when flagged, with a "check-in edited since" note.

**Notes / edge cases**
- This is the explainability slice that earns manager trust — it directly supports retention priority (1).
- Highlight spans come from S1's detector output; if the model can't attribute a span, fall back to showing the full check-in with no highlight rather than a wrong one.

**Out of scope**
- Editing or dismissing the flag. Model retraining from this view.

---

### S4 — Dismiss a flag that's wrong

As an **engineering manager**,
I want to dismiss an at-risk flag I disagree with,
so that my digest stays trusted and we stop showing me noise.

**Acceptance criteria**
- Given a flagged check-in, when I dismiss the flag, then it disappears from my current digest view and is recorded as dismissed.
- Given I dismissed a flag, when the same check-in would be re-flagged in a later digest, then it stays dismissed and does not reappear.
- Given I dismiss a flag, when the action is recorded, then the dismissal (with the check-in id and reason if given) is captured as a labeled negative for future eval-set refresh.
- Given a network failure on dismiss, when I retry, then the dismissal is idempotent — dismissing twice has the same effect as once.

**Notes / edge cases**
- Dismissals feed the feedback loop: confirmed-noise becomes training/eval signal (handed to Priya for `blocker-eval-v2`).
- Split deliberately from S5 (confirm) — happy-path dismiss ships before positive confirmation.

**Out of scope**
- Confirming a flag as correct (S5). Auto-tuning the model live. Per-team retraining.

---

### S5 — Confirm a flag is a real blocker

As an **engineering manager**,
I want to confirm a flag is a genuine blocker,
so that I can act on it and the team gets credit for a true signal.

**Acceptance criteria**
- Given a flagged check-in, when I confirm it as a real blocker, then it's marked confirmed in the digest and recorded as a confirmed positive.
- Given a confirmed flag, when next week's check-in still references the same blocked work, then the digest shows "still at risk (confirmed N days ago)".
- Given I confirm a flag, when the action is recorded, then it's captured as a labeled positive for eval-set refresh.

**Notes / edge cases**
- Confirm + dismiss together close the human-feedback loop the analyst needs for v2 of the eval set.
- One verb per story: this is "confirm" only; dismiss is S4.

**Out of scope**
- Assigning the blocker to someone, creating a task, or any workflow action beyond marking it.

---

### S6 — Tune detector sensitivity per team

As an **engineering manager**,
I want to set how sensitive the Blocker Detector is for my team,
so that a chatty team isn't drowned in flags and a terse team still gets caught.

**Acceptance criteria**
- Given I open Blocker Detector settings, when I choose a sensitivity (Low / Balanced / High), then the next digest applies that threshold for my team only.
- Given I haven't set anything, when the detector runs, then it uses the default (Balanced) threshold.
- Given I set sensitivity to Low, when the detector runs, then only flags above the high-confidence threshold appear (fewer flags, higher precision).
- Given an invalid value is submitted (out-of-range via API), when it's saved, then it's rejected and the previous valid setting is retained.

**Notes / edge cases**
- Maps to the precision/recall trade-off from S1 — Low favors precision, High favors recall.
- Business-rule-variation split: this is the per-team rule isolated from core detection.

**Out of scope**
- Per-person sensitivity. ML auto-tuning. Org-wide defaults set by an admin (later).

---

### S7 — Notify the team member when their check-in is flagged

As a **team member writing a check-in**,
I want to know when my own check-in was flagged at-risk,
so that I can add context before my manager asks, instead of being surprised in the digest.

**Acceptance criteria**
- Given my check-in was flagged at-risk, when the digest is generated, then I receive a quiet in-app notice that my check-in was flagged, with the reason.
- Given I have notifications muted, when my check-in is flagged, then I get no notice and the flag still appears in my manager's digest.
- Given I add a follow-up comment after being notified, when the manager opens the flag, then my comment appears alongside the flag.

**Notes / edge cases**
- Supports retention (1): teams that feel surveilled stop checking in. Framing must be supportive, not punitive — copy reviewed by Maya.
- Notice is async/in-app only at launch; no email or Slack push yet.

**Out of scope**
- Email/Slack delivery. Letting the member suppress the manager-side flag.

---

### S8 — Track Blocker Detector accuracy in production

As the **product team**,
I want a weekly view of detector precision/recall in production using confirm/dismiss feedback,
so that we catch drift and decide when to refresh the eval set and retune.

**Acceptance criteria**
- Given a week of confirm/dismiss feedback, when the weekly accuracy job runs, then it reports observed precision and recall against confirmed/dismissed labels.
- Given production precision drops below 0.65 in a week, when the job runs, then it raises an alert to the product team.
- Given a week with fewer than 30 labeled feedback events, when the job runs, then it reports "insufficient data" rather than a misleadingly precise number.

**Notes / edge cases**
- Closes the loop opened by S4/S5. Owned jointly by PM + analyst.
- Deferred-performance / measurement slice — it makes the feature trustworthy over time, not at first ship.

**Out of scope**
- Automated retraining. Live model swaps. Per-team accuracy breakdown.

## Suggested build order

1. **S1 — Detect against the eval set (offline)** — thinnest end-to-end slice; proves the signal is real before any UI. Everything visible to teams gates on this clearing the bar.
2. **S2 — Surface flags in the digest** — first user-visible value; gets the signal in front of leads.
3. **S3 — See why a check-in was flagged** — earns manager trust, the lever for retention priority (1).
4. **S4 — Dismiss a wrong flag** — protects digest trust and starts the feedback loop (negative labels).
5. **S5 — Confirm a real blocker** — completes the human-feedback loop (positive labels) the analyst needs.
6. **S7 — Notify the flagged team member** — keeps the feature supportive so teams keep checking in.
7. **S6 — Tune sensitivity per team** — reduces noise once we have enough teams using it; depends on the trade-off being understood.
8. **S8 — Track accuracy in production** — measurement slice; needs S4/S5 feedback volume to be meaningful.

**Sprint note (🟡 at risk):** With one engineer on PTO 3 days, a realistic sprint is **S1 + S2**. S1 is uncertain (model iteration may not clear the bar first pass), so we treat S2 as a stretch, not a commitment. S3–S8 are next-sprint candidates.
